### Failure modes Formula E v1 exposed (durably recorded)

1. **Shared-document host gaming** — `docs.google.com`, `eu1.make.com`, `drive.google.com` shared links counted as "distinct hosts." Not the builder's owned/controlled artifacts. (Olalekan case.)

2. **Free-platform subdomain breadth gaming** — 12 `*.vercel.app` deploys count as 12 distinct hosts. Should count as ~1 effective platform. (Ryan case.)

3. **Multi-TLD same product** — `mailiac.com` + `mailiac.camera` count as 2 hosts but are one product. (Jan case.)

4. **Dead L1 receipts still scoring as L1** — l1_ratio doesn't decay when artifacts die. (Ryan, Jan, Emanuel cases.)

5. **Atlas confidence captures surface, not substance** — a 2-commit GitHub repo and a 200-commit production repo can score similar confidence. (Olalekan vs Aniket comparison.)

6. **AI-generated repos pass L1** — large day-1 commits + polished README + no commit-pattern over time = current classifier scores them as real. (General 2026 pattern, research-validated.)

### Research findings (load-bearing, recorded for future batches)

**Verification ladder L0–L5** (replaces existing 5-level model; richer + better-aligned with industry):

| Layer | Verifies | Source | Status |
|---|---|---|---|
| L0: Claimed | Builder says they shipped X | Builder | BUILT |
| L1: Reachable | URL responds, classifier sees something | Engine HTTP | BUILT (gameable, see failure modes) |
| L2: Substantive | Artifact has depth (commits over time, code complexity, content density, browser-tested demo) | Engine deeper-crawl | DESIGNED, not built |
| L3: Attested | Buyer/hackathon/employer entity confirms via cross-attestation | Network entity graph | DESIGNED (Path B / D2-D3), not built |
| L4: Economic | Real money/transactions/usage logs (Stripe, on-chain, RentAHuman, Moltbook agent activity) | External economic systems | NOT DESIGNED — 2026 differentiator opportunity |
| L5: Signed | W3C Verifiable Credential cryptographically signed by reputable third party | Cryptographic | DESIGNED (current L4_cryptographically_signed in schema), long-term |

### Architectural principle to record durably

**Payment-as-filter, not payment-as-rank.** Surfaced when operator asked whether paid profiles should rank higher. Research conclusion (sourced from X Premium / Upwork Boosted / Upwork Top-Rated / Fiverr Pro / Toptal comparison): payment may unlock features and visibility surface area (custom URL, multiple atlas roles, featured eligibility, Pro badge), but payment MUST NOT multiply or override the engine-derived quality_score in default ranking. X Premium destroyed its trust signal exactly because it merged payment with the verified badge. ShipStacked's open-access + machine-verified-proof positioning depends on quality_score remaining sacrosanct.

### Self-tagged context at intake (designed, not built)

Operator-proposed mechanism: builder tags each Build Feed item with context — Personal experiment / Personal live / Team-org / Paying customer / Hackathon submission / Open source. Context multiplier weights each receipt. High-weight claims (paying customer, hackathon winner) require attestation via the entity graph (Buyer entity confirms, hackathon entity links). This is a cheaper near-term substitute for full L2 substance verification — relies on self-report + lie-cost-via-network-attestation. Recorded for future implementation; defers to entity graph (Path B) shipping first.
